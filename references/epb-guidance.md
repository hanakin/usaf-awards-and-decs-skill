# Performance Report Guidance

Use this reference for performance-report statement work after reading `statement-shared.md`. In this skill, `EPB` and `OPB` are treated as interchangeable terms unless the user later requires a distinction.

## Performance-report rules

- Use ALQ-based routing: `E`, `L`, `M`, `I`, and `H`
- Keep the member name near the front when the format expects named statements
- Keep statements in the correct section instead of blending multiple ALQs without a reason
- Preserve plain-English readability over compressed board slang
- Treat approved example language as valid guidance for acceptable EPB style in this skill
- Treat `H` as a push statement that can be more recommendation-oriented than `E`, `L`, `M`, and `I`
- For `H`, it is acceptable to emphasize readiness for greater responsibility when that matches the requested style and approved examples

## Hard length rules

- `E`, `L`, `M`, and `I`: `350` characters
- `H`: `250` characters

Count the entire string, including letters, numbers, whitespace, punctuation, symbols, and special characters.

## Guardrails

- Do not force `1206`-style dash bullets into a performance report unless the provided format requires it
- Do not append quality-ranking suffixes unless the requested output explicitly uses them
- Keep mandatory comments and evaluation requirements from `AFI 36-2406` separate from statement advocacy
- Do not flatten `H` into the same tone as the other ALQs; preserve its push-statement role

## Workflow

1. Review the provided achievements and determine which ALQ category fits best
2. Confirm the target section before drafting
3. Build one statement at a time for that section
4. Generate variations for that single statement when the user asks
5. Validate exact count, acronym use, spelling, grammar, clarity, and section fit before delivery

Use `statement-validation.md` for the detailed validation order, troubleshooting logic, and conflict-resolution priority.

Use `performance-report-examples.md` to compare ALQ fit, one-statement-at-a-time patterns, and approved `H` push-statement style before drafting.
